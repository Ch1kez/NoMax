## AGENTS.md

Правила и подсказки для нейросетей/агентов, работающих с проектом NoMax.

### Общие принципы

- **Сохраняй работоспособный dev‑контур**:
  - не ломай `docker-compose.yml`, `Dockerfile`, `Dockerfile.frontend`, `infra/livekit/livekit.yaml`;
  - после изменений код должен подниматься командой `docker compose up -d --build`.
- **Не завязывайся на внешние SaaS‑медиа‑сервисы**:
  - медиа‑часть должна работать через self‑hosted LiveKit (сервис `livekit` в `docker-compose.yml`);
  - не добавляй Twilio/Daily/Zoom и т.п. без явного запроса.
- **Старайся не усложнять стек**:
  - бекенд остаётся на FastAPI;
  - фронтенд пока — простой HTML/JS PWA через Nginx; если добавляешь React/TS, делай это как слой поверх, не ломая существующий поток.

### Бекенд

- Файлы:
  - `main.py` — маршруты FastAPI, CORS, WebSocket‑сигналинг.
  - `auth.py` — JWT, OAuth2PasswordBearer, логика логина.
  - `models.py` — доменные модели (`User`, `Relationship`, `CallRoom`) и in‑memory хранилище (для MVP).
  - `media.py` — интеграция с LiveKit (чтение `LIVEKIT_*` env, генерация JWT).
- При добавлении эндпоинтов:
  - не ломай существующие URL и схемы ответов (лучше расширяй, чем меняй);
  - обязательно учитывай авторизацию (`Depends(get_current_user)` там, где нужно).
- Если потребуется вынести хранение из памяти в БД:
  - добавляй слой репозиториев/ORM, но не меняй публичные модели и контракт API без крайней необходимости.

### LiveKit и WebRTC

- LiveKit‑сервер должен оставаться **self‑hosted**:
  - не меняй `LIVEKIT_HOST` на облачный URL без прямого запроса пользователя;
  - при изменении конфига следи, чтобы фронт и бекенд использовали один и тот же URL/ключи.
- При доработке WebRTC‑логики:
  - не шифруй/логируй медиа‑потоки на бекенде;
  - не добавляй запись звонков без отдельного раздела в README и согласования с пользователем.

### Фронтенд

- Текущий фронт — одна страница `frontend/index.html` с минимальным JS:
  - не удаляй существующие UX‑флоу (логин → список близких → звонок);
  - если добавляешь состояние/логику, старайся оставить интерфейс чистым и простым.
- При добавлении React/TS:
  - не ломай текущий PWA‑shell; по возможности делай React‑бандл, который монтируется внутрь существующей разметки;
  - не добавляй тяжёлые сборщики без необходимости (Vite / esbuild предпочтительнее тяжелого Webpack).

### Docker / инфраструктура

- Любые изменения в Docker:
  - должны позволять запуск полной системы через одну команду `docker compose up -d --build`;
  - не должны требовать ручной установки зависимостей в системе пользователя.
- Если меняешь порты / env:
  - обнови README, чтобы инструкция по запуску всегда была актуальна;
  - не используй нестандартные порты, уже занятые системой, без необходимости.

### Документация

- При существенных изменениях (новые эндпоинты, шаги запуска, новые фичи):
  - обновляй `README.md` (коротко и по делу);
  - не переносись в отдельные громоздкие документы, если это можно описать в 2–3 разделах README.
